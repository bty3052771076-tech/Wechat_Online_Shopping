## 数据库概述

### 基本信息

| 项目 | 值 |
|------|-----|
| **数据库名** | `wechat_shop` |
| **字符集** | `utf8mb4` |
| **排序规则** | `utf8mb4_unicode_ci` |
| **时区** | `Asia/Shanghai` |
| **引擎** | `InnoDB` |
| **表数量** | 24张 |
| **预估数据量** | 小型（< 10万条记录） |

### 设计原则

1. **遵循三范式**：消除数据冗余，避免更新异常
2. **适当反范式**：对于频繁查询的字段允许冗余（如订单商品快照）
3. **使用软删除**：重要数据使用status字段标记删除，而非物理删除
4. **时间戳统一**：所有表使用created_at和updated_at记录时间
5. **金额存储**：所有金额字段使用DECIMAL(10,2)，单位为元
6. **主键策略**：使用BIGINT自增主键

## 数据库规范

### 命名规范

#### 表命名
- 全部小写
- 使用下划线分隔
- 复数形式（users, orders, products）
- 示例：`user_addresses`, `order_items`

#### 字段命名
- 全部小写
- 使用下划线分隔
- 见名知意
- 布尔字段使用is_前缀
- 示例：`user_id`, `is_default`, `created_at`

#### 索引命名
- 主键索引：`PRIMARY`
- 唯一索引：`uk_字段名`（unique key）
- 普通索引：`idx_字段名`（index）
- 示例：`uk_username`, `idx_user_id`

### 数据类型规范

| 类型 | 用途 | 示例 |
|------|------|------|
| `BIGINT` | 主键、外键、ID | `id`, `user_id` |
| `VARCHAR(50)` | 短文本（名称、编码） | `username`, `phone` |
| `VARCHAR(200)` | 中等文本（标题、地址） | `title`, `address` |
| `VARCHAR(500)` | 长文本（URL、描述） | `image_url`, `detail` |
| `TEXT` | 超长文本（详情、HTML） | `product_detail` |
| `DECIMAL(10,2)` | 金额（精确到分） | `price`, `total_amount` |
| `INT` | 数量、序号 | `quantity`, `stock` |
| `TINYINT` | 状态、类型、枚举 | `status`, `gender` |
| `DATETIME` | 时间戳 | `created_at`, `updated_at` |
| `DATE` | 日期 | `production_date` |
| `JSON` | JSON数据（MySQL 5.7+） | `specs`, `images` |

### 字段约束规范

- **主键**：`NOT NULL AUTO_INCREMENT`
- **外键**：建立索引，不强制外键约束（提高性能）
- **非空**：重要字段设置为`NOT NULL`
- **默认值**：设置合理的DEFAULT值
- **注释**：所有字段必须添加COMMENT

## 索引设计

### 索引策略

1. **主键索引**：所有表都有自增主键
2. **唯一索引**：username, email, phone, order_no等唯一字段
3. **普通索引**：频繁查询的字段（user_id, status, created_at）
4. **复合索引**：多字段组合查询（user_id + sku_id）

### 索引优化建议

- ✅ 为WHERE条件字段添加索引
- ✅ 为JOIN字段添加索引
- ✅ 为ORDER BY字段添加索引
- ❌ 避免过度索引（影响写入性能）
- ❌ 低选择性字段不建索引（如gender）
